# 校正模块重设计方案

## 现状问题

当前校正功能集中在 `MainWindow::openCalibrationDialog()` 和 `MainWindow::sendCalibrationSquareGrid()` 中，UI、图案生成、坐标编码、TCP 下发、日志和安全校验耦合在一个函数里。它现在只能发送固定 3x3 正方形阵列，缺少校正数据采集、误差求解、参数保存和回放验证，因此更像“校正图案发送器”，还不是完整校正模块。

现有实现还直接复用加工发送器 `machining_data_buffer_`，这会让校正、手动点动和加工共享同一套队列状态。后续如果加工中断、暂停、急停或 TCP 反压，校正模块很难独立判断自己的发送生命周期。

## 设计目标

1. 校正流程独立于加工流程，拥有自己的状态机和发送队列。
2. 图案生成、数据帧编码、TCP 发送、误差求解、配置保存分层实现。
3. 支持“只预览/不开光测试/开光加工/采集测量点/拟合参数/验证图案”完整闭环。
4. 所有校正结果可追溯：保存输入图案、测量点、拟合残差、坐标变换矩阵、版本和时间戳。
5. 校正参数进入统一坐标变换链路，而不是散落在 UI 或发送函数里。

## 模块划分

### CalibrationPattern

负责生成逻辑校正图案，不涉及 TCP 和设备状态。

- 输入：图案类型、中心、间距、数量、尺寸、Z 高度、扫描速度。
- 输出：逻辑路径段 `CalibrationPath`，包含 MARK/JUMP 段、图案点编号、期望几何位置。
- 第一阶段支持：3x3 方格、十字靶标、同心圆、外框矩形。

### CalibrationEncoder

负责把逻辑路径转为板卡 16 字节帧。

- 使用统一的轴配置、Y/Z 交换、A/B 固定策略、延迟参数。
- 输出帧流和元数据：总帧数、预计执行时间、每个靶标的帧范围。
- 禁止在编码层直接访问 UI 控件。

### CalibrationTransport

负责校正专用下发。

- 使用独立 `DataBuffer`，不复用 `machining_data_buffer_`。
- 提供 `start/send/stop/laserOff/waitTransportDrained`。
- 暂停或急停时先停止旧队列，再发送关光帧。
- 日志记录队列状态和板卡反馈，但不把 `waitUntilIdle()` 当作物理完成。

### CalibrationSolver

负责根据测量结果求解校正参数。

- 输入：图案理论点、用户/相机测得点。
- 输出：平移、旋转、缩放、剪切或二阶畸变参数。
- 第一阶段可先做 2D 仿射：`machine_xy -> measured_xy`。
- 第二阶段扩展到曲面加工时的局部平面姿态和五轴补偿。

### CalibrationStore

负责保存和加载校正配置。

- 建议保存到 `config/calibration/*.json`。
- 内容包括设备、轴模式、图案参数、测量点、拟合模型、残差、创建时间。
- UI 只选择“当前启用的校正配置”，加工和手动点动统一从配置读取。

## 推荐状态机

`Idle -> PreviewReady -> Sending -> WaitingExecution -> Measurement -> Solved -> Verified -> Applied`

异常分支：

- `Sending/WaitingExecution -> EmergencyStop -> LaserOffSent -> Idle`
- `Measurement/Solved -> Discarded -> Idle`

UI 上不要显示“校正完成”，除非已经完成测量和求解。单纯帧下发完成只能显示“图案已下发”。

## 实施步骤

1. 先把现有 `sendCalibrationSquareGrid()` 拆成 `CalibrationPattern` 和 `CalibrationEncoder`，行为保持不变。
2. 新增校正专用 `DataBuffer`，校正发送不再使用 `machining_data_buffer_`。
3. 给校正发送加关光和停止路径，复用加工侧的安全策略。
4. 新增测量点表格 UI：理论点、测量点、误差、是否参与拟合。
5. 实现 2D 仿射求解和 JSON 保存。
6. 在手动点动、校正、加工三条链路统一调用同一个坐标校正入口。

## 当前优先级建议

优先做 1-3。这样能先把安全性和模块边界理顺，不会影响当前加工主流程。求解和保存可以在图案发送稳定后再做。
